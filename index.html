Асинхронное программирование и распределенное кэширование

Ниже приведены результаты теста сравнения оптимальной скорости при использовании асинхронного программирования и распределенного кэширования.


MVC-паттерн

Сравните скорость asp.net mvc с веб-формой:

Name Time (ms)
Webform 70ms
ASP MVC 10ms

ASP.NET MVC показывает пустое представление не более чем за 7 миллисекунд (но при первой загрузке приложения, времени компиляции плюс время рендеринга) и Интернет-формы отображать ту же пустую форму не менее чем за 70 миллисекунд. Даже если абсолютное значение бесполезно, относительное значение является доказательством очень разного минимального значения тайм-аута для двух конструкций. ASP.NET MVC почти в 10 раз лучше веб-форм!

Распределенное кэширование

Ниже показана оптимизация скорости с использованием распределенного кэширования для веб-приложения ASP.NET.


Результаты сравнения скорости:

Name Time (ms)
NoCache 314ms
Cache 37ms

Когда нами было запущено приложение без использования кэша потребовалось 314мс, чтобы ответить на 406 сообщений в базе данных. При использовании кэша тот же объем данных был получен за 37мс. Быстродействие сайта увеличилось более чем в 10 раз по сравнению с тем, когда не используется распределенное кэширование.

DbContextPool

результаты тестов при использовании dbcontextpool для оптимизации скорости приложений .NET Core

Пример кода можно посмотреть по 2 ссылкам ниже: docker pull bao2976/dbcontextpoolv1
https://github.com/itmo308555/test_dbcontextpool

Для оценки скорости обработки запросов DbContext и DbContextPool создается небольшой тест. Для измерения производительности используется инструмент Apache Bechmarking Tool. Первый эксперимент заключается в том, что в 10 объектов отправляется по 100 запросов. Выполнение скрипта показано на рисунке 9, результаты показаны в таблице 1.

Name Time take for tests (s) Requests per sencond (s) Time per Request (s)
DbContext 4.71 21.23 47.101
DbContextPool 3.896 25.57 38.96

Данные, приведенные в таблице 1, показывают, что разница между двумя методами невелика.





Во втором эксперименте количество запросов увеличивается до 10.000, а количество объектов до 100. Выполнение скрипта показано на рисунке 10, результаты показаны в таблице 2.

Name Time take for tests (s) Requests per sencond (s) Time per Request (s)
DbContext 570.071 17.54 67.007
DbContextPool 478.484 20.57 47.848

Данные, приведенные в таблице 2, показывают, что разница между двумя методами существенна по сравнению с таблицей 1. Используя DbContextPooling, количество и время отклика программы довольно хорошо повышают производительность. Среднее количество запросов в секунду приложение будет обрабатывать быстрее, чем раньше. Это очень маленькое приложение, и результаты, вероятно, будут значительно отличаться при использовании в более крупном приложении.

ИСПОЛЬЗОВАНИЕ gRPC

Сравнение производительности gRPC с Rest API

Пример кода можно посмотреть по 2 ссылкам ниже: docker pull bao2976/codingmilitiarestvsgrpcsamplebenchmark
https://github.com/itmo308555/gRPCvsRest

Эксперимент 1:

В этом эксперименте мы создали 2 сервера: один - gRPC (с использованием Protobuf), а другой - с использованием REST. Оба сервера отвечают ключевым словом «Hello Word». Мы запустили тест на обоих серверах с 1000 входящих подключений к каждому серверу.

Method Time (ms)
Rest 387.9
gRPC 438.0

Результаты показывают, что при нормальном доступе к серверу Rest, кажется, немного быстрее, чем gRPC: 387,9 мс для Rest и 438,0 мс для gRPC.

Эксперимент 2:

В этом тесте мы продолжаем создавать 2 сервера, один - gRPC (с использованием Protobuf), а другой - с использованием REST. Но здесь мы попросили сервер делать случайные ответы на 1000 последовательных запросов к серверу (т.е. генерировать большую загрузку страницы). Результаты были разные!

Method Time (ms)
Rest 4.277
Grpc 1.952

При большой загрузке веб-сайта, т.е. при большом количестве запросов на выполнение команд к серверу, отдых был намного медленнее, чем gRPC: 4277 с по сравнению с 1 952 с.


Этот тест показывает, что gRPC быстрее. Это связано с тем, что gRPC обычно использует буферы протокола. (Это механизм для сериализации структурированных данных), тогда как REST обычно отправляет данные в XML или JSON. gRPC использует HTTP/ 2 для поддержки масштабируемых и высокопроизводительных API-интерфейсов и использует двоичные данные вместо простого текста, что делает взаимодействие более компактным и эффективным.
